# Ontology Engine

The `@openfoundry/engine` package is the **operational core** of the Open Foundry platform. It manages the full lifecycle of ontology objects and links, enforces validation rules, emits lifecycle events (as CNCF CloudEvents), evaluates computed fields, tracks data lineage, and manages persistent object sets. It sits directly above the SPI and below the action and API layers, composing the [[spi]] and [[odl]] packages to provide rich domain operations that the upper layers depend on.

## Public API

**Object Lifecycle:**
- `ObjectManager` — Creates, reads, updates, soft-deletes, and hard-deletes ontology objects through the SPI. Applies property validation via `validateObjectProperties` against ODL schema definitions before write. Accepts `ObjectManagerConfig`.
- `validateObjectProperties(schema, objectDef, properties)` → `ValidationResult` — Validates property types, required fields, enum values, constraint directives.
- `validationError(message, field)` → `ValidationFailure` — Utility for structured validation errors.

**Link Management:**
- `LinkManager` — Creates, reads, updates, and deletes ontology links through the SPI. Generates UUIDv7 link IDs before storage. Enforces cardinality constraints (ONE_TO_ONE, ONE_TO_MANY, MANY_TO_ONE, MANY_TO_MANY). Accepts `LinkManagerConfig`.
- `generateUUIDv7()` — Time-ordered unique ID generator for link primary keys.

**Event System:**
- `EventBus` (interface) — Pluggable event bus for ontology mutation events.
- `InMemoryEventBus` — Default in-memory pub/sub implementation.
- `EngineEventEmitter` — Wraps an `EventBus` and emits typed `CloudEvent` objects on object/link mutations.
- `EventCause`, `ChangeSet`, `ObjectEventData`, `LinkEventData` — Event payload types.

**Computed Fields:**
- `ComputedFieldEvaluator` — Evaluates `@computed` fields using registered compute functions. Supports custom `ComputeFunction` implementations. Accepts `ComputedFieldEvaluatorConfig`.
- `ComputeContext` — The context passed to compute functions (object data, schema info).

**Data Lineage:**
- `LineageRecorder` — Records field-level provenance when objects are created or mutated. Supports querying lineage history.
- `InMemoryLineageStore` — In-memory implementation of `LineageStore`.
- `LineageQueryOptions` — Filter/sort criteria for lineage queries.

**Object Sets:**
- `ObjectSetManager` — Manages named, persistent collections of objects for batch operations, supporting Create/Read/Update/Delete and membership queries.
- `InMemoryObjectSetStore` — In-memory implementation of `ObjectSetStore`.

## Dependencies

- **`@openfoundry/spi`** — All operations flow through `StorageProvider`.
- **`@openfoundry/odl`** — Schema definitions drive validation and computed field evaluation.
- **`@openfoundry/observability`** — OpenTelemetry tracing and structured logging.

Dev dependencies: `@openfoundry/storage-memory` (test SPI backend).

## Used By

- [[api-gateway]] — GraphQL resolvers use `ObjectManager` and `LinkManager` for reads; action executions call through [[action-executor]] which uses `ObjectManager`/`LinkManager` for writes.
- [[action-executor]] — The `ActionExecutor` invokes `ObjectManager` and `LinkManager` to apply action effects within a transaction.

## Key Design Decisions

- **UUIDv7 link IDs** — Link IDs are generated by the Engine (not the SPI) using UUIDv7 (time-ordered, globally unique). This ensures consistent ID generation regardless of storage backend and enables ordered traversal. SPI providers MUST NOT override link IDs.
- **Computed field architecture** — `@computed` fields are evaluated lazily by the `ComputedFieldEvaluator`. Computed values are cached (with TTL) and invalidated when dependencies change. This keeps business logic in the engine layer, not the database.
- **Lineage vs Audit** — [[ontology-engine]] tracks **data lineage** (where data came from, field-level provenance). [[security-service]] tracks **audit** (who did what, when). These are complementary but distinct concerns: lineage is about data quality and trust; audit is about accountability and compliance.
